# Gitブランチモデル解説レポート

## 1. はじめに：ブランチモデルとは何か？

結論から言うと、どのブランチモデルを採用しても、使用するソフトウェアはすべて同じ「**Git**」です。

「Git-flow」「GitHub Flow」「GitLab Flow」といった名称の違いは、別のソフトが必要という意味ではなく、**「Gitという道具をどう使うかという『ルール（運用方針）』の違い」**を指します。

料理に例えると、以下のようなイメージです。

* **Git** = 包丁（道具）
* **Git-flow / GitHub Flow** = 野菜の切り方（千切りにするか、乱切りにするかという流派・ルール）

運用設計においては、エンジニアではない関係者にもこの違いを理解してもらい、プロジェクトに最適な「切り方（ルール）」を選定することが重要です。

---

## 2. なぜ「GitHub」「GitLab」という名前がついているのか？

これらの名称は、その運用ルールを誰が考案・提唱したかに由来しています。

* **Git-flow**:
  2010年にVincent Driessen氏という個人がブログで提案した、「元祖」となる厳格なルールです。
* **GitHub Flow**:
  GitHub社が提唱。「Git-flowは複雑すぎる！Web開発ならもっとシンプルでいい！」という思想で作られたルールです。
* **GitLab Flow**:
  GitLab社が提唱。「GitHub Flowはシンプルすぎて、企業の実務（検証環境など）に対応しきれない！」という課題を解決するために作られたルールです。

**重要**: 「GitHub上でGit-flow」を行うことも、「GitLab上でGitHub Flow」を行うことも可能です。あくまで「チームで決めたルール」の話です。

---

## 3. 3つの代表的なブランチモデル比較

それぞれの決定的な違いは、**「本番環境（ユーザーが見ている画面）に反映させるまでの『慎重さ』」**にあります。

### ① Git-flow（超慎重・重厚）

**「定期的なリリース日（発売日）が決まっているアプリ向け」**

* **特徴**: 非常に厳格で複雑。
* **構造**:
  * 開発用 (`develop`) と本番用 (`master`) が完全に分かれています。
  * その他に `feature` (機能開発), `release` (リリース準備), `hotfix` (緊急修正) が存在します。
* **流れ**:
  1. 普段は `develop` で開発する。
  2. リリース日が来たら、`develop` から `release` ブランチを作り、最終テストする。
  3. 完璧になったら `master` に合流させ、バージョン番号をつけて公開する。
* **イメージ**: 電車のダイヤ改正や、スマホアプリのアップデート。準備期間を経て「せーの」で公開するスタイルです。

### ② GitHub Flow（超高速・シンプル）

**「毎日何度も更新するWebサービス向け」**

* **特徴**: 非常にシンプル。現在のWeb開発の主流です。
* **構造**: 本番用 (`main`) 一本だけが中心です。
* **流れ**:
  1. `main` から作業用ブランチを切る。
  2. 作業が終わったらレビューを受ける。
  3. OKなら `main` に合流させ、**その瞬間に本番環境へ自動反映（デプロイ）**する。
* **イメージ**: ニュースサイトやSNS。記事や機能を修正したら、即座にユーザーに見せるスタイルです。

### ③ GitLab Flow（現実的・バランス型）

**「Webサービスだけど、一旦テスト環境で確認したい企業向け」**

* **特徴**: GitHub FlowとGit-flowの中間。
* **構造**: GitHub Flowに加え、「環境ごとのブランチ（`production`, `staging`）」を持ちます。
* **流れ**:
  1. 開発はGitHub Flowと同じように進める。
  2. `main` に合流させた後、すぐ本番には出さない。
  3. まず `staging`（検証環境）ブランチに反映させてテストする。
  4. 問題なければ `production`（本番環境）ブランチに反映させて公開する。
* **イメージ**: 企業の業務システム。バグが出ると困るので、本番公開前に「テストサーバー」で確認するワンクッションを置くスタイルです。

## 4. まとめ：判断基準

どのブランチモデルを採用するか決定する際の基準は以下の通りです。

| フロー名 | こんなプロジェクトに最適 | キーワード |
| :--- | :--- | :--- |
| **GitHub Flow** | スタートアップ、小規模チーム、Webサービス。<br>「スピード命！完成したらすぐ出したい！」 | **シンプル・最速** |
| **GitLab Flow** | 受託開発、中規模以上の企業、業務システム。<br>「スピードは大事だけど、客先確認（検証環境）が必要」 | **実務的・環境別** |
| **Git-flow** | インストール型ソフト、スマホアプリ、金融系。<br>「バグは許されない。リリース日が決まっている」 | **厳格・品質重視** |

### 結論
「ソフトウェアが違う」わけではないので、**プロジェクトの性質（スピード重視か、品質重視か）**に合わせて、チームで話し合って「今回はこの流派でいこう」と決めるのが、運用設計の第一歩です。
